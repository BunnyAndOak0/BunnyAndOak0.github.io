Servlet知识点的整理

什么是servlet

​ servlet 是运行在 Web 服务器中的小型 Java 程序（即：服务器端的小应用程序）。servlet 通常通过 HTTP（超文本传输协议）接收和响应来自 Web 客户端的请求。

执行过程

生命周期

实例化 –> 初始化 –> 服务 –>销毁

出生：（实例化 –> 初始化）第一次访问Servlet就出生（默认情况下）

活着：应用存在就活着，servlet也就活着

死亡：应用卸载servlet也就销毁（卸载不是结束程序，二者之间不一样）

Servlet生命周期的方法

  1. 实例化

    只有在第一次访问时才被调用

    public ServletDemo1(){
	System.out.println("***********ServletDemo1执行了*********");
}

  2. 初始化

    只有在第一次访问的时候才被调用

    public void init(ServletConfig arg0) throws ServletException {
	System.out.println("***********init执行了*********");
}

  3. 服务

    每次访问都会被调用

    public void service(ServletRequest arg0, ServletResponse arg1)
		throws ServletException, IOException {
	//System.out.println("hello servlet");
	System.out.println("***********service执行了*********");
}

  4. 销毁

    在被销毁时调用

    public void destroy() {
	System.out.println("***********destroy执行了*********");
}

Tips:

​ 使Servlet在服务器启动的时候就创建，配置<load-on-startup>2</load-on-startup>可以使得Servlet在服务器启动的时候就创建，也分为不同的情况：
（其中的数字一定要为整数，从小到大，数字越小等级越高，也就越先启动，数字相同的话启动顺序随机，0为最小，但是不建议使用，因为在默认中使用的就是0） 当<load-on-startup>所设的数值 >= 0的时候，表示容器在启动的时候就加载并初始化这个Servlet
​ 当<load-on-startup>所设的数值 < 0的时候，表示该Servlet被调用的时候才会被加载和初始化

转发

有两种方式获得转发对象(RequestDispatcher)：

  1. 一种是通过HttpServletRequest的getRequestDispatcher()方法获得。
  2. 一种是通过ServletContext的getRequestDispatcher()方法获得；

RequestDispatcher getRequestDispatcher(String path) ;//参数表示要跳转到哪去。

例如：

request.getRequestDispatcher ("demo.jsp"). forward(request, response);			//转发到demo.jsp

转发对于浏览器来说是透明的，也就说只发出了一个请求，获得了一个响应，浏览器的url不变。

重定向

HttpServletResponse的sendRedirect()方法。

服务器根据此请求寻找资源并发送给客户，它可以重定向到任意URL，不能共享request范围内的数据。

例如：

response.sendRedirect("demo.jsp");	//重定向到demo.jsp

此时，浏览器请求重新定位，浏览器的URL会改变。

重定向与转发的主要区别

  1. sendRedirect()方法不但可以在位于同一个主机上的不同Web应用程序之间进行重定向，而且可以将客户端重定向到其他服务器上的Web应用程序资源。而forward（）方法只能将请求转发给同一Web应用的组件。

  2. sendRedirect()方法不能转发到“/WEB-INF/”路径下的jsp页面资源，而getRequestDispatcher().forword()可以重定向到“/WEB-INF/”路径下的jsp页面资源。如 request.getRequestDispatcher(“/WEB-INF/jsp/login.jsp”).forward(request, response);

    WEB-INF是Java的WEB应用的安全目录。所谓安全就是客户端无法访问，只有服务端可以访问的目录。而重定向就相当于用户直接从客户端访问了的路径，自然就不可以啦，只有程序内部转发的时候才能转发到WEB-INF下的资源。

  3. 转发：浏览器URL的地址栏不变（一次请求过程）。

    重定向：浏览器URL的地址栏改变（两次请求过程）。

过滤器

介绍：

  1. 来自Servlet规范中的一个接口
  2. 可以在请求到达目标资源文件之前进行拦截，从而判断本次要求是否合法，或者对拦截请求进行增强
  3. 过滤器接口实现需要由开发人员负责实现

实现步骤：

  1. 创建一个Filter接口实现类
  2. 重写doFilter方法
    • 进行判断
    • 或者增强
  3. web.xml注册过滤器
    • 设置刚被拦截的请求

过滤器的配置

<!-- 过滤器 -->
    <filter>
        <filter-name>encoding</filter-name> <!-- 设置过滤器名称 以便mapping匹配过滤器 -->
        <filter-class>com.cn.fanjg.servlet.EncodingServlet</filter-class><!-- 过滤器实现类 -->
        <init-param><!-- 初始化参数 可以在FilterConfig中读取该参数 -->
            <param-name>encoding</param-name>
            <param-value>utf-8</param-value>
        </init-param>    
    </filter>
    <!-- 过滤器匹配条件 -->
    <filter-mapping>
        <filter-name>encoding</filter-name>
        <url-pattern>/*</url-pattern>
    </filter-mapping>

实现代码：

public class EncodingServlet extends HttpServlet implements Filter {
    private FilterConfig config = null;
    private String encoding = "";

    @Override
    public void doFilter(ServletRequest request, ServletResponse response,
            FilterChain chain) throws IOException, ServletException {
        HttpServletResponse resp = (HttpServletResponse) response;
        resp.setCharacterEncoding(encoding);
        java.io.PrintWriter print = response.getWriter();
        print.print("过滤器设置编码为：" + this.encoding);
        chain.doFilter(request, response);
    }

    @Override 
    public void init(FilterConfig config) throws ServletException {
        this.config = config;
        this.encoding = config.getInitParameter("encoding");
    }
}

FilterChain接口

  1. 接口实现类由Tomcat负责提供
  2. 本质上是一个数组
  3. 数组存放对本次请求要求拦截的所有过滤器对象
  4. 过滤器链条对象相当于一个调度，决定哪一个过滤器先执行，哪一个过滤器后执行

监听器

域对象：在某一个范围之内，可以为Servlet之间提供共享数据的对象。例如：

  1. 全局作用域对象：ServletContext application，在Tomcat运行期间，可以为当前工程中的所有Servlet提供共享数据。
  2. 会话作用域对象：HttpSession session，在一次会话过程中，为参与本次会话的Servlet提供共享数据。
  3. 请求作用于对象：HttpServletRequest request，在一次请求处理过程中，为参与本次请求的所有Servlet提供共享数据，比如：请求转发。

监听器接口：专名监听域对象生命周，期变化以及域对象共享数据的变化情况。

cookie

（会话跟踪, 也就是一个数据的载体）

会话：就是用户打开浏览器，从发第一次请求开始，一直到关闭浏览器，就是一种会话

​ Cookie由服务器生成，保存在客户端的一种信息的载体，这个载体中，存放着用户访问该站点的会话状态信息。由若干键值对构成。用户第一次提交请求后，由服务器生成，并封装到响应头，以响应的形式发送给客户端。如果再次发送同类请求，就会在请求中携带Cookie的数据。

不同浏览器的cookie的存放位置不同，因此删除一个浏览器上的，不会影响其他的。

例如：

public class Cookie_Test extends HttpServlet {
	@Override
	protected void doGet(HttpServletRequest req, HttpServletResponse resp) throws ServletException, IOException {
		Cookie cookie = new Cookie("Company", "lanmei");
		Cookie cookie2 = new Cookie("teacher", "wangwang");
		
		//指定绑定路径  需要添加项目名称  只是写到缓存中
		cookie.setPath(req.getContextPath() + "/aaa/bbb");
		cookie2.setPath(req.getContextPath() + "/ccc/ddd");
		
		/*设置Cooki额、的有效期，是一个整型值，单位为秒
		大于0表示存放至硬盘
		小于0，与不设置相同，是放入缓存
		等于0，表示和Cookie一生成，马上失效*/
		cookie.setMaxAge(60 * 60);    //有效一小时
		cookie2.setMaxAge(60 * 60 * 24 * 10); 		//有效期十天
		
		resp.addCookie(cookie);
		resp.addCookie(cookie2);
	}
}

session

Session基本用法

​ 也就是会话，是web开发中的一种会话状态跟踪技术。Cookie也是一种会话跟踪技术，但是cookie是将状态保留在客户端，而Session是将会话状态保留在服务器端。

​ 会话：就是用户打开浏览器，从发第一次请求开始，一直到关闭浏览器，就是一种会话。

Session的用法

一般情况下，如果要向Session中写入数据，需要使用getSession（true），也就是getSession（）方法

而如果要从Session中读取数据，需要使用getSession（false），具体见代码

例如：

@WebServlet("/SomeServlet")
public class SomeServlet extends HttpServlet {
	protected void doPost(HttpServletRequest request, HttpServletResponse response) throws ServletException, IOException {
		String username = request.getParameter("username");
		//将参数放入request域中  Otherservlet 获取不到数值
		request.setAttribute("user", username);
		
		//获取Session对象   OtherServlet可以获取数据  true表示 有session就用旧的session没有session就创建一个新的session，不填的话默认为utrue
		HttpSession session = request.getSession(true);
		session.setAttribute("user", username);
		
		response.getWriter().print("SomeServlet" + username);
	}
}

		String user = request.getParameter("user");
		response.getWriter().print("OtherServlet_user：" + user);
		
		//从Session中获取数据  可以获取到SomeServlet的数据
		//获取Session   false表示有session就用旧的session 没有的话也不重新创建
		HttpSession session = request.getSession(false);
		String username = null;
		if (session != null) {
			username = (String)session.getAttribute("user");
		}
		response.getWriter().print("OtherServlet_username：" + username);
	}
}

Session工作原理

  1. 写入Session列表

    ​ 当用户第一次提交请求时，服务端Servlet会执行request.getSession()方法后，会自动生成一个Map.Entry对象，key是一个根据某种算法新生成的JSessionID，value则为新创建的HttpSession对象

    (Session列表)

  1. 服务器生成并发送Cookie

    ​ 以“JSessionID”作为name，value值为32为JSessionID的值的字符串，包装成一个Cookie放到响应头中，发送给客户端

  1. 客户端接受并发送Cookie

    ​ 接收到服务器端发送过来的Cookie后会保存至客户端的缓存中，当客户端再次发送请求时，会将浏览器缓存中的Cookie,放入请求的头部信息中，发送给服务器。

  1. 从Session列表中查找

    ​ 服务端从请求中读取客户端发送来的Cookie的JSessionID的值， 从Map中查找相应的value，也就是Session的对象，然后对该Session对象的域属性进行读写操作。

Session的失效

默认为30分钟，从最后一次被访问开始计时。

<session-config>
	<session-timeout>120</session-timeout>
</session-config>

可以设置失效时间，单位为分钟

但是Session失效之后，并不为null，失效之后，解绑在上面的所有数据。（解绑可以使用 removeAttribute（XXX）来实现）

Cookie禁用后重定向跳转时Session的跟踪

String uri = request.getContextPaht() + "/otherServlet";
uri = response.encodeRedirectURL(uri);	   //进行重新编码 这样会自动在地址后面加上jsessionid=XXX这样的值
response.sendRedirect(uri);

但是这样的话，jsessionid会被写道地址栏上，这样就不安全

Cookie禁用后非重定向跳转时Session的跟踪

response.setContentType("text/html;charset=utf-8");
PrintWriter out = response.getWriter();
String uri = "otherServlet";
//解决Cookie禁用后，非重定向时的Sessionn跟踪问题
uri = response.encodeURL(uri);
out.println("<a href = '" + uri +"'>跳转</a>到OtherServlet");

三个与属性空间的对比

作用范围由大到小：

ServletContext —> HttpSession —> HttpServletRequest

ServletContext：全局，整个应用范围，是跨会话的共享数据

HttpSession：域属性是会话范围，可以完成跨请求的共享数据

HttpServletRequest：域属性是请求范围，可以完成跨Servlet共享数据，但是这些Servlet必须在同一个请求中。

​ 在可以保证需求的前提下，优先使用范围小的，这样可以节省服务器的内存，还可以保证数据的安全性。
